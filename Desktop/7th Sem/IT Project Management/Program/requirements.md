Restaurant Name: Presto
### Menu Display
Story 1. Instant Menu Access via QR Code
As a Restaurant Customer, I want to scan a QR code with my phone, so that I can instantly access the restaurant's menu.
Acceptance Criteria
A.C.1.1 Given the customer scans the QR code, when their phone's camera recognizes it, then they are taken directly to the web-based menu.
A.C.1.2 The QR code is tested and confirmed to be functional, linking to the correct deployed URL.

Story 2. Mobile-Optimized Menu Readability
As a Restaurant Customer, I want the menu to be readable and easy to navigate on my mobile phone, so that I don't have to zoom or scroll horizontally.
Acceptance Criteria
A.C.2.1 Given the customer is on a mobile device, when they open the menu, then the website is responsive.
A.C.2.2 All text, images, and layout elements fit the screen width without requiring horizontal scrolling.
Story 3. Detailed Menu Item Display
As a Restaurant Customer, I want to browse all menu items, so that I can see the item's name, price, description, and a picture.
Acceptance Criteria
A.C.3.1 Given the customer is on the menu page, when they look at an item, then they can see its Name, Image, Price, and Description.
A.C.3.2 The system is confirmed to be view-only; no "order" or "add to cart" buttons are present, as this is out of scope.
Story 4. Welcoming Homepage Confirmation
As a Restaurant Customer, I want to see a welcoming Homepage, so that I can confirm I'm viewing the correct restaurant's menu.
Acceptance Criteria
A.C.4.1 Given the customer lands on the Homepage, when the page loads, then they see the restaurant's name and logo.
A.C.4.2 A welcome message is displayed as specified in the project's design mockups.
Story 5. Visibility of Specials Section
As a Restaurant Customer, I want to see a 'Specials' section, so that I can know what the current promotions are.
Acceptance Criteria
A.C.5.1 Given the customer is on the menu page, when they browse, then a "Specials" section is clearly visible.
A.C.5.2 The content in the "Specials" section matches the content manually updated in the code by the Project Manager.
Story 6. Location Map Access (Customer)
As a Restaurant Customer, I want to view the 'Location Map', so that I can easily find the café's address.
Acceptance Criteria
A.C.6.1 Given the customer navigates to the "Location Map" page, when the page loads, then an embedded Google Map is displayed.
A.C.6.2 The map correctly shows the restaurant's pinned location as specified by the client.

Story 7. Contact Information Access (Customer)
As a Restaurant Customer, I want to find the 'Contact' information, so that I can call the restaurant or see their details.
Acceptance Criteria
A.C.7.1 Given the customer navigates to the "Contact" page, when the page loads, then they can see the restaurant's contact details.
A.C.7.2 The displayed details (e.g., phone, address) are accurate per client-provided information.
Story 8. About Us Page Access (Customer)
As a Restaurant Customer, I want to read the 'About Us' page, so that I can learn more about the café's story.
Acceptance Criteria
A.C.8.1 Given the customer navigates to the "About Us" page, when the page loads, then they can read the restaurant's background information.
A.C.8.2 The content displayed on the page matches the "About Us" text provided by the client.
Story 9. QR Code Image Deliverable (Client)
As the Client, I want to receive a high-resolution QR code image, so that I can print it for my café tables.
Acceptance Criteria
A.C.9.1 Given the project is complete, when deliverables are handed over, then a high-resolution QR code image is provided as a deliverable.
A.C.9.2 The delivered QR code image is tested and verified to link to the live, functional menu.

Story 10. Project Source Code Ownership (Client)
As the Client, I want to receive the complete project source code, so that I own the final product.
Acceptance Criteria
A.C.10.1 Given the project is complete, when deliverables are handed over, then the full project source code is provided.
A.C.10.2 The source code is delivered in a standard format (e.g., .zip file) and is complete.
Story 11. Fixed Project Deadline (Client)
As the Client, I want the project completed by the fixed deadline, so that I can meet my academic and business goals.
Acceptance Criteria
A.C.11.1 Given the project is in progress, when the date is 30th November 2025, then all development and User Acceptance Testing are complete.
A.C.11.2 The final project submission is made to the client and executive sponsor by the end date of 30th November 2025.
Story 12. Manual Specials Update (Project Manager)
As the Project Manager, I need to manually update the 'Specials' section in the code, so that I can fulfill the requirement without building a CMS.
Acceptance Criteria
A.C.12.1 Given the client requests an update to "Specials," when the PM edits the source code, then the live menu's "Specials" section is updated.
A.C.12.2 This process is confirmed to work while adhering to the out-of-scope constraint of "no CMS".
